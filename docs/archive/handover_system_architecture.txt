       구성:
       [Raw 보존층]
         └─ Parquet 파일 (로컬 FS 또는 MinIO)
              - 원본 무결성 유지
              - DuckDB ad-hoc 분석 가능

         └─ PostgreSQL 16 + pgvector
              ├─ schema: metadata    (데이터셋·컬럼·제약)

              ├─ schema: ml_results  (모델 학습·SHAP·예측)
              ├─ schema: agent_logs  (Agent 의사결정 기록)
              ├─ schema: lineage     (변환 체인 추적)
              └─ schema: rag_vectors (pgvector — 도메인 문서 임베딩)

       선택 사유:
       - 한국 SI 시장 ★압도적 표준★ (고객사 거의 100% 이미 운영 중)
       - 도입 장벽 최저
       - 운영 노하우·백업·HA·모니터링 도구 성숙
       - 트랜잭션 ACID 안전성
       - pgvector 로 RAG 통합 → ★단일 DB 운영★
       - 옵션 B (DuckDB+Qdrant+MinIO) 와 옵션 C (TimescaleDB) 검토 후 기각

★ S11. Harness Engineering 4요소

       Harness = LLM Agent 를 안전·효과적으로 운영하기 위한 인프라 묶음

       ① 도구 스키마 검증
          MCP 도구 입출력 JSON Schema 강제 검증
          → LLM 의 잘못된 호출 사전 차단

       ② 컨텍스트 관리
          큰 데이터는 LLM 컨텍스트에 직접 넣지 않음
          → MCP 가 ★요약·샘플링★ 만 전달, RAG 로 보강

       ③ 가드레일 (S8 의 3-tier 권한 모델)

       ④ Lineage 추적 (★보완 추가 요소)
          모든 변환을 schema:lineage 에 기록
          → 재현 가능성 + 책임 추적 + SI 컴플라이언스
          → DVC/MLflow/LakeFS 같은 도구 차용 가능

────────────────────────────────────────────────────────────────────────────────
[9-1] 4단 A2A 구조 (S9 재진술)
────────────────────────────────────────────────────────────────────────────────

  Inspector
    역할: raw 데이터 받아 메타데이터·스키마·인코딩·기초 통계 추출
    권한: L1 자동
    출력: DataProfile (JSON)

  Planner
    역할: DataProfile + 사용자 제약 → 전처리 시퀀스 계획
    권한: 계획만 (실행 권한 없음)
    출력: PreprocessingPlan (JSON, 각 단계의 권한 등급 포함)

  Executor
    역할: PreprocessingPlan 의 각 단계 실행
    권한: 단계별 분기 (L1 즉시 / L2 승인 대기 / L3 차단)
    출력: ProcessingResult (변환된 데이터셋 + Lineage)

  Validator
    역할: ProcessingResult 검증 + Lineage 기록 + 다음 단계 라우팅
    권한: 검증만
    출력: ValidationReport + DB 기록
